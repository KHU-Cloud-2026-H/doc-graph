# DocGraph 아키텍처 설계

## 문서 목적

이 문서는 DocGraph MVP의 상위 아키텍처를 정의한다. 기능 요구사항과 비기능 요구사항을 어떤 시스템 구성으로 만족시킬 것인지 설명하며, 상세 API 명세나 DB 스키마 수준의 구현 설계는 이후 상세 설계 문서에서 다룬다.

즉, 본 문서는 다음 질문에 답한다.

- 시스템을 어떤 컴포넌트로 나눌 것인가
- 각 컴포넌트에 어떤 AWS 서비스를 배치할 것인가
- 왜 해당 컴퓨팅 모델을 선택했는가
- WAF 기준에서 이 구조가 어떤 근거를 가지는가

## 설계 원칙

- AWS를 필수 클라우드로 사용한다.
- AWS Well-Architected Framework의 5개 Pillar를 기준으로 설계한다.
- 워크로드 성격에 따라 적절한 컴퓨팅 모델을 분리한다.
- WAF 관점에서 설명되지 않는 AWS 서비스는 도입하지 않는다.

## 설계 범위

본 문서는 다음 범위를 포함한다.

- 프론트엔드 호스팅 구조
- API 서버 및 Webhook 수신 구조
- 정합성 검증 비동기 파이프라인
- 데이터 저장소 구성
- 운영 및 개발 환경의 기본 방향

본 문서는 다음 범위를 포함하지 않는다.

- 상세 API 스펙
- DB 테이블 및 컬럼 설계
- 프롬프트 구조와 AI 응답 계약
- 세부 예외 처리 로직

## 컴퓨팅 모델 분리

DocGraph는 모든 작업을 하나의 실행 모델에 몰아넣지 않는다. 사용자 요청 처리, Webhook 수신, AI 기반 정합성 검증은 성격이 다르므로 각 영역에 맞는 실행 모델을 선택한다.

| 영역 | 성격 | 선택 모델 | 선택 이유 |
| --- | --- | --- | --- |
| API 서버 | 상시 실행, 복잡한 비즈니스 로직 | Spring Boot on ECS/Fargate | 복잡한 도메인 로직 관리가 쉽고, OAuth 및 인증 처리에 유리하며, 콜드 스타트 없는 일관된 응답 시간을 제공한다 |
| 변경 감지 | 이벤트 기반, 빠른 응답 필요 | Spring Boot on ECS/Fargate | Webhook 수신 후 빠르게 의존 쌍을 조회하고 큐잉해야 하며, 이미 상시 실행 중인 서버를 활용할 수 있다 |
| 정합성 검증 | 느리고 비용이 큰 비동기 작업 | AWS Lambda | SQS 트리거 기반 처리에 적합하고, AI 호출 타임아웃과 실패를 별도 격리할 수 있다 |

## 전체 아키텍처 개요

### 사용자 요청 경로

```text
[사용자]
    ↓
CloudFront + S3
    ↓
API Gateway
    ↓
Spring Boot on ECS/Fargate
  - Notion OAuth 연동
  - 의존 관계 그래프 조회 및 관리
  - 충돌 해소 API
  - 결정사항 조회 API
    ↓
RDS PostgreSQL
```

### 이벤트 처리 경로

```text
Notion Webhook
    ↓
Spring Boot Webhook 엔드포인트
  - 변경 이벤트 수신
  - Notion API 호출로 최신 페이지 내용 조회
  - 의존 관계 쌍 조회
  - SQS에 검증 작업 큐잉
  - 즉시 200 응답 반환
    ↓
SQS
    ↓
Lambda 정합성 검증 워커
  - AI API 호출
  - 충돌 구간, 원인, 수정 제안 생성
  - 결과 저장
  - Notion 코멘트 삽입
  - 외부 Webhook 알림 발송
    ↓
RDS PostgreSQL
```

## 컴포넌트별 역할

### 1. 프론트엔드

- React 기반 웹 애플리케이션을 사용한다.
- 정적 자산은 S3에 배포하고 CloudFront를 통해 제공한다.
- 주요 화면은 프로젝트 목록, 그래프 조회, 위반 항목 조회, 결정사항 조회를 포함한다.

### 2. API Gateway

- 외부 공개 진입점 역할을 담당한다.
- 사용자 API 요청을 ECS/Fargate 기반 Spring Boot 서비스로 전달한다.
- 공개 HTTPS 엔드포인트를 제공한다.

### 3. Spring Boot on ECS/Fargate

상시 실행되는 핵심 애플리케이션 계층이다. 다음 역할을 담당한다.

- Notion OAuth 연동
- 프로젝트 생성 및 설정 관리
- 문서 메타데이터 조회 및 그래프 조회 API 제공
- 정합성 위반 조회 및 해소 처리 API 제공
- Notion Webhook 수신
- 변경 문서 기준 의존 관계 쌍 조회 및 SQS 큐잉

이 계층은 사용자 요청과 빠른 이벤트 수신 처리를 담당하며, 느린 AI 호출은 직접 수행하지 않는다.

### 4. SQS

- 정합성 검증 작업의 비동기 버퍼 역할을 한다.
- Webhook 수신 경로와 AI 기반 검증 경로를 분리한다.
- 일시적 부하 증가 시 검증 작업을 흡수하는 완충 계층 역할을 한다.
- 실패 작업 보존 및 재처리 설계를 지원한다.

### 5. Lambda 정합성 검증 워커

- SQS 메시지를 트리거로 실행된다.
- 연결된 문서 쌍을 기준으로 AI API를 호출해 정합성 위반 여부를 판단한다.
- 충돌 구간, 원인, 수정 제안을 생성해 저장한다.
- 필요 시 Notion 코멘트 자동 삽입과 외부 Webhook 알림 발송을 수행한다.

이 계층은 느리고 실패 가능성이 높은 외부 호출을 격리하는 역할을 가진다.

### 6. RDS PostgreSQL

DocGraph의 주요 상태 저장소다. 다음 데이터를 저장한다.

- 프로젝트 및 사용자 연동 정보
- 문서 메타데이터
- 문서 타입 분류 결과
- 의존 관계 그래프 정보
- 결정사항 추출 결과
- 정합성 위반 이력 및 해소 상태

## 데이터 흐름 요약

### 문서 변경 감지부터 위반 표시까지

1. Notion이 페이지 변경 이벤트를 Webhook으로 전송한다.
2. Spring Boot 서비스가 이벤트를 수신한다.
3. Spring Boot 서비스가 Notion API를 호출해 최신 페이지 내용을 가져온다.
4. 변경된 문서와 연결된 의존 관계 쌍을 조회한다.
5. 각 문서 쌍을 SQS에 비동기로 등록한다.
6. Lambda가 각 메시지를 처리하며 AI API를 호출한다.
7. 위반이 감지되면 결과를 저장하고 그래프 상태를 갱신한다.
8. Notion 코멘트 삽입 및 외부 Webhook 알림을 시도한다.
9. 사용자는 인앱 화면에서 위반 상태와 수정 제안을 확인한다.

### 충돌 해소 흐름

1. 사용자가 서비스에서 위반 항목을 해소 처리한다.
2. Spring Boot 서비스가 위반 상태를 갱신한다.
3. 그래프에서 해당 엣지의 상태가 즉시 변경된다.

## 기술 스택

| 레이어 | 기술 | 비고 |
| --- | --- | --- |
| 프론트엔드 | React | 세부 프레임워크는 추후 확정 |
| 프론트 호스팅 | S3 + CloudFront | 정적 웹 자산 배포 |
| API 서버 | Spring Boot | Docker 컨테이너화 후 ECS/Fargate 배포 |
| Webhook 수신 | Spring Boot | Notion Webhook 수신, 의존 쌍 조회, SQS 큐잉 |
| 비동기 처리 | AWS Lambda | SQS 트리거 기반 정합성 검증 |
| 메시지 큐 | AWS SQS | 비동기 버퍼 및 실패 격리 |
| 데이터베이스 | RDS PostgreSQL | 그래프 구조, 문서 메타데이터, 위반 이력 저장 |
| 문서 소스 | Notion API + Notion Webhook | OAuth 연동 및 변경 이벤트 수신 |
| AI 호출 | 외부 AI API | 공급자는 추후 확정 |
| 알림 | 인앱 표시 + 외부 Webhook | Slack, Discord 등 호환 |
| 컨테이너 오케스트레이션 | ECS/Fargate | EC2 직접 운영 제거 |

## WAF 관점 정리

| Pillar | 적용 내용 |
| --- | --- |
| Operational Excellence | Notion Webhook을 기반으로 변경 이벤트를 자동 수신해 별도 스케줄링 없이 운영 흐름을 단순화한다 |
| Reliability | SQS와 비동기 워커로 실패를 격리하고, 개별 검증 실패가 전체 파이프라인을 중단시키지 않도록 설계한다 |
| Security | OAuth 토큰, 데이터베이스 자격증명, 외부 API 키 등 민감 정보를 보호하고 승인된 애플리케이션 컴포넌트만 데이터 저장소에 접근하도록 제한한다 |
| Performance Efficiency | 사용자 요청 처리와 AI 검증 처리를 분리해 빠른 응답이 필요한 경로와 느린 비동기 경로를 각각 최적화한다 |
| Cost Optimization | AI 호출은 변경된 문서 쌍에 대해서만 수행하고, 비동기 워커는 이벤트 기반으로 실행해 유휴 비용을 줄인다 |

## 개발 환경

개발 및 데모 환경은 로컬 검증과 AWS 배포를 분리하는 방식으로 운영한다.

### 역할 분리

- 인프라 담당은 Terraform으로 환경을 구성한다.
- 백엔드 담당은 로컬 환경에서 E2E 테스트를 통과시키는 것을 우선 목표로 한다.
- 이후 인프라 담당이 AWS Learner Lab에 배포한다.

### 로컬 에뮬레이션 대상

| 대상 | 로컬 구성 | 용도 |
| --- | --- | --- |
| Lambda | LocalStack Lambda | 정합성 검증 함수 로컬 실행 |
| SQS | LocalStack SQS | 비동기 처리 흐름 로컬 테스트 |

### 개발 흐름

```text
인프라 담당
  Terraform으로 LocalStack 환경 구성
      ↓
백엔드 담당
  Spring Boot + Lambda 로직 개발
  docker-compose + LocalStack으로 E2E 테스트
  Notion Webhook은 ngrok 등으로 로컬 수신
      ↓
인프라 담당
  E2E 통과 코드를 AWS Learner Lab에 배포
```

### 데모 대비 fallback

LocalStack에서 AWS Learner Lab로 이전이 원활하지 않을 경우, LocalStack 기반 환경을 데모 목적으로 유지하는 대안을 고려한다.

## 문서 위치

본 문서는 상위 아키텍처 설계 문서다. 이후 단계에서는 다음 항목을 상세 설계로 확장해야 한다.

- AI 검증 요청 및 응답 포맷
