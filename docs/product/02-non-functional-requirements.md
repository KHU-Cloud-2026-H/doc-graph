# DocGraph 비기능 요구사항

## 문서 목적

이 문서는 DocGraph MVP가 충족해야 하는 비기능 요구사항을 정의한다. 기능 요구사항이 시스템이 무엇을 해야 하는지를 설명한다면, 비기능 요구사항은 시스템이 어느 수준의 신뢰성, 성능, 보안성, 운영 가능성을 갖추어야 하는지를 규정한다.

아키텍처 설계 문서는 이러한 기준을 만족시키기 위한 기술적 선택을 설명하고, 본 문서는 그 선택이 만족해야 하는 품질 기준 자체를 정의한다.

## 적용 범위

- Notion 연동 및 동기화 흐름
- 의존 관계 그래프 생성 및 조회
- 정합성 검증 파이프라인
- Notion 코멘트 삽입
- 외부 Webhook 알림

## 신뢰성(Reliability)

| 항목 | 요구사항 |
| --- | --- |
| 비동기 작업 재처리 | 정합성 검증 작업은 일시적 실패 발생 시 자동 재시도를 지원해야 한다 |
| 실패 격리 | 한 문서 쌍의 정합성 검증 실패가 다른 문서 쌍의 처리에 영향을 주지 않아야 한다 |
| 재처리 안전성 | 동일 이벤트가 재처리되더라도 동일 Notion 코멘트가 중복 삽입되지 않아야 한다 |
| 알림 실패 폴백 | 외부 Webhook 알림 발송에 실패해도 인앱 위반 표시는 유지되어야 한다 |
| Notion 반영 실패 폴백 | Notion 코멘트 삽입에 실패해도 인앱 위반 표시는 유지되어야 한다 |
| 실패 보존 | 반복 실패한 작업은 유실되지 않고 후속 분석 및 재처리가 가능해야 한다 |

### 신뢰성 해설

- 정합성 검증은 비동기 처리이므로 개별 작업 단위의 실패를 분리할 수 있어야 한다.
- 중복 이벤트나 재시도는 운영 환경에서 자연스럽게 발생할 수 있으므로 idempotency가 필요하다.
- 외부 시스템 연동 실패는 사용자에게 위반 사실을 숨기는 이유가 되어서는 안 된다.

## 성능(Performance)

| 항목 | 요구사항 |
| --- | --- |
| Webhook 수신 응답 시간 | Notion Webhook 수신 후 서비스는 3초 이내에 응답을 반환해야 한다 |
| 정합성 검증 AI 타임아웃 | 단일 정합성 검증 요청의 AI 응답 대기 시간은 30초를 초과하지 않아야 한다 |
| 충돌 해소 반영 시간 | 사용자가 위반 항목을 해소 처리하면 그래프 상태는 즉시 반영되어야 한다 |
| 그래프 조회 반응성 | 사용자는 프로젝트 전환 및 그래프 조회 시 체감 가능한 지연 없이 결과를 확인할 수 있어야 한다 |

### 성능 해설

- Webhook 엔드포인트는 무거운 후처리를 직접 수행하지 않고 빠르게 수신 확인을 반환해야 한다.
- AI 호출은 전체 처리 시간의 병목이 될 수 있으므로 명시적 타임아웃이 필요하다.
- 해소 상태 반영은 사용자 인터랙션이므로 비동기 배치가 아니라 즉시성이 중요하다.

## 보안(Security)

| 항목 | 요구사항 |
| --- | --- |
| OAuth 토큰 보호 | 사용자별 Notion OAuth 토큰은 암호화된 상태로 저장되어야 한다 |
| 비밀정보 관리 | 데이터베이스 비밀번호, AI API 키 등 시스템 자격증명은 안전한 비밀 저장소를 통해 관리되어야 한다 |
| 데이터 접근 통제 | 데이터베이스는 승인된 애플리케이션 컴포넌트만 접근 가능해야 한다 |
| 전송 구간 보호 | 외부 연동 및 사용자 요청은 암호화된 HTTPS 연결을 사용해야 한다 |
| 최소 권한 원칙 | 각 실행 컴포넌트는 자신의 역할 수행에 필요한 최소 권한만 가져야 한다 |

### 보안 해설

- 본 문서는 보안 목표를 정의하며, 구체적으로 어떤 저장소나 네트워크 구성을 사용할지는 아키텍처 설계 문서에서 다룬다.
- 특히 OAuth 토큰과 외부 API 키는 서비스 신뢰성뿐 아니라 사고 대응 비용에도 직접적인 영향을 준다.

## 운영 가능성(Operability)

| 항목 | 요구사항 |
| --- | --- |
| Webhook 엔드포인트 가용성 | Notion Webhook을 수신하는 공개 HTTPS 엔드포인트는 안정적으로 운영되어야 한다 |
| 장애 가시성 | Webhook 수신 실패, AI 검증 실패, Notion 코멘트 삽입 실패는 운영자가 식별할 수 있어야 한다 |
| 로그 추적성 | 각 처리 단계는 원인 추적이 가능하도록 요청 단위 로그를 남겨야 한다 |
| 재처리 가능성 | 실패한 비동기 작업은 운영자가 재처리할 수 있어야 한다 |
| 상태 확인 가능성 | 프로젝트별 위반 항목과 처리 상태를 운영 및 사용자 관점에서 확인할 수 있어야 한다 |

### 운영 해설

- 이 시스템은 외부 서비스에 강하게 의존하므로, 실패 여부를 보지 못하면 사실상 운영이 불가능하다.
- 로그와 상태 추적이 없다면 정합성 위반이 검출되지 않은 것인지, 검출됐지만 반영이 실패한 것인지 구분할 수 없다.

## 설계 연계 메모

- 재시도 횟수, DLQ 사용 여부, 비밀 저장소 종류, 네트워크 격리 방식은 아키텍처 설계 문서에서 구체화한다.
- 본 문서의 기준은 설계 선택의 근거가 되며, 특정 클라우드 서비스나 구현체에 종속되지 않도록 유지한다.

## 요약

DocGraph의 비기능 요구사항 핵심은 다음과 같다.

- 외부 연동 실패가 발생해도 위반 상태는 유실되지 않아야 한다.
- Webhook 수신과 사용자 상호작용은 빠르게 응답해야 한다.
- OAuth 토큰과 시스템 자격증명은 안전하게 보호되어야 한다.
- 운영자는 실패 원인을 추적하고 재처리할 수 있어야 한다.
