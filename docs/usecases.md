# 유즈케이스

사용자 행위 중심으로 주요 시나리오를 정의한다.

---

## UC1. 회원가입 및 워크스페이스 연결

**액터**: 워크스페이스 소유자

**시나리오**
1. 사용자가 DocGraph에 접속해 "Notion으로 로그인"을 선택한다.
2. Notion OAuth 인증 화면에서 연결할 워크스페이스를 선택하고 권한을 승인한다.
3. DocGraph에 계정이 생성되고 해당 Notion 워크스페이스가 연결된다.
4. 인박스 화면으로 이동한다.

**사후 조건**: 워크스페이스 소유자로 워크스페이스가 등록되어 있다.

---

## UC2. 프로젝트 생성 및 초기 동기화

**액터**: 워크스페이스 소유자

**전제 조건**: UC1 완료

**시나리오**
1. 워크스페이스 목록에서 워크스페이스를 선택해 프로젝트 목록으로 이동한다.
2. "새 프로젝트"를 선택하면 Notion 루트 페이지 목록이 표시된다.
3. 프로젝트로 사용할 루트 페이지를 선택한다.
4. 하위 페이지 트리가 로드되고, 각 상위 페이지에 문서 타입을 매핑한다.
5. 프로젝트 멤버를 초대하고 역할(Admin/Member)을 부여한다.
6. 타입별 담당자 기본값을 설정한다.
7. "동기화 시작"을 눌러 초기 동기화를 트리거한다. 루트 페이지 하위 트리 전체를 가져온다.
8. 동기화 완료 후 문서 그래프 화면으로 이동한다. 엣지와 연결 제안이 생성되어 있다.

**사후 조건**: 프로젝트가 생성되고 문서·엣지·연결 제안이 초기화되어 있다.

---

## UC3. 문서 변경 감지 및 충돌 알림 수신

**액터**: 멤버 (Notion에서 문서를 수정), 담당자 (알림 수신)

**전제 조건**: UC2 완료, 해당 문서에 의존 관계 엣지가 존재

**시나리오**
1. 멤버가 Notion에서 프로젝트 내 문서를 수정하고 저장한다.
2. DocGraph가 Webhook 이벤트를 수신해 변경된 페이지 내용을 조회한다.
3. 임베딩이 재생성되고, Notion 링크 diff가 계산되어 엣지가 갱신된다.
4. 영향받는 엣지에 대해 AI 정합성 검증이 실행된다.
5. 충돌이 감지되면 엣지가 빨간색으로 표시되고, target 문서 담당자의 인박스에 항목이 추가된다.
6. 프로젝트에 Webhook URL이 설정된 경우 Slack/Discord로 알림이 발송된다.

**사후 조건**: 충돌 항목이 인박스에 표시되어 있다.

---

## UC4. 인박스에서 충돌 확인 및 무시 처리

**액터**: 담당자 (멤버 또는 Project Admin)

**전제 조건**: UC3 완료, 인박스에 미해소 충돌 항목이 존재

**시나리오**
1. 사용자가 인박스를 열어 자신에게 할당된 미해소 충돌 목록을 확인한다.
2. 충돌 항목을 선택하면 문서 비교 패널이 열린다.
3. 충돌 구간, 원인, AI 수정 제안을 검토한다.
4. 수정이 필요 없다고 판단하면 "무시"로 마킹한다.
5. 해당 항목이 인박스에서 사라지고 엣지는 무시됨으로 표시된다. Conflict에 ignoredAt·ignoredBy가 기록된다.
6. 이후 해당 문서 쌍 중 하나가 외부에서 변경되면 무시 처리가 자동으로 해제되어 재검증된다.

**사후 조건**: 충돌 항목에 ignoredAt·ignoredBy가 기록되어 있다.

## UC5. 연결 제안 수락

**액터**: Project Admin

**전제 조건**: UC2 완료, 문서 그래프에 연결 제안(점선 엣지)이 존재

**시나리오**
1. Admin이 문서 그래프에서 점선 엣지를 클릭한다.
2. 이분할 뷰로 두 문서 내용이 표시된다.
3. Admin이 내용을 검토한 후 "수락"을 선택한다.
4. 점선 엣지가 실선으로 전환되고 정합성 검증이 즉시 트리거된다.
5. 검증 결과에 따라 엣지 상태가 업데이트된다.

**사후 조건**: 새 DependencyEdge가 생성되고 정합성 검증 결과가 반영되어 있다.

## UC6. 수동 엣지 추가

**액터**: Project Admin

**전제 조건**: UC2 완료

**시나리오**
1. Admin이 문서 그래프에서 출발 문서 노드를 선택한다.
2. 문서 패널의 의존 관계 목록에서 "추가"를 선택한다.
3. 도착 문서를 검색해 선택하고 체크 항목을 입력한다.
4. 엣지가 생성되고 정합성 검증이 즉시 트리거된다.

**사후 조건**: 커스텀 DependencyEdge가 생성되고 정합성 검증 결과가 반영되어 있다.

---

## UC7. 담당자 설정

**액터**: Project Admin

**전제 조건**: UC1 완료, 멤버가 프로젝트에 배정되어 있음

**시나리오**
1. 프로젝트 설정에서 타입별 담당자 기본값을 멤버 단위로 지정한다.
2. 이후 해당 타입 문서가 target인 충돌이 감지되면 지정된 담당자의 인박스에 항목이 추가된다.

**사후 조건**: 타입별 담당자가 설정되어 충돌 항목이 올바른 인박스로 라우팅된다.

## 프론트 연동 참고사항

- **프로젝트 생성 플로우**: `POST /workspaces/{id}/projects` → 타입 매핑·담당자·멤버 설정 → `POST /projects/{id}/sync` 순서로 호출한다. sync는 설정 완료 후 수동으로 트리거한다.
- **그래프 화면 데이터**: 초기 렌더링은 `GET /projects/{id}/graph` 하나로 처리한다(`nodes`, `edges`, `proposals` 포함). 패널·관리 목록이 필요할 때만 개별 list API(`/edges`, `/proposals`, `/documents`)를 사용한다.
